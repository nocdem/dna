# Stage F — Multi-Node Integration Harness

3-node localhost cluster for testing consensus-affecting changes
**before** production deploy. Catches state_root divergence bugs that
unit tests miss (unit tests are single-node; consensus bugs live in
multi-node state).

## Why

Every consensus fix during 2026-04-19 session (txn_id, attendance v1,
attendance v2) passed unit tests but failed live:

- Multi-node state_root divergence can't be modeled in a single-node
  test.
- Production cluster wipe per test = 10+ min per iteration; harness
  iteration = seconds.

Stage F was in the original hard-fork v1 plan and was skipped. This is
the retrofit.

## Layout

All runtime state under `/tmp/stagef-$TIMESTAMP/` — **fully isolated**
from production:

- `/tmp/stagef-*/node[1-3]/identity/` — fresh Dilithium5 identities,
  auto-generated by nodus-server on first run. NOT the production
  keys in `/var/lib/nodus/identity/`.
- `/tmp/stagef-*/node[1-3]/data/` — witness DB + nodus DB + logs.
- `/tmp/stagef-*/user/.dna/` — fresh dna wallet for the test
  operator. NOT punk's `~/.dna/`. Isolated via `HOME=...`.

## Port map

Each node uses a 10-port stride starting at 14000. Disjoint from
production (4000-4004) so both can run simultaneously.

| Node | UDP | TCP client | TCP peer | TCP chan | TCP witness |
|---|---|---|---|---|---|
| 1 | 14000 | 14001 | 14002 | 14003 | 14004 |
| 2 | 14010 | 14011 | 14012 | 14013 | 14014 |
| ... | | | | | |
| 7 | 14060 | 14061 | 14062 | 14063 | 14064 |

## Identity isolation

**NEVER reads or writes:**
- `/var/lib/nodus/*` (production witness/identity)
- `~/.dna/*` (punk's production wallet)
- ports 4000-4004 (production)

## Usage

```
# Build nodus + messenger first
cd /opt/dna/nodus/build && make -j$(nproc)
cd /opt/dna/messenger/build && make -j$(nproc)

# Bring up 3-node cluster + submit genesis
bash /opt/dna/nodus/tests/integration/stagef/stagef_up.sh

# Run a consensus test (checks state_root identical 3/3)
bash /opt/dna/nodus/tests/integration/stagef/tests/test_undelegate.sh

# Teardown
bash /opt/dna/nodus/tests/integration/stagef/stagef_down.sh
```

## Scripts

| Script | Purpose |
|---|---|
| `stagef_up.sh` | Generate identities + spawn 3 nodus-server + wait peer mesh + submit genesis + fund user |
| `stagef_down.sh` | Kill PIDs + rm -rf the run dir |
| `stagef_diff.sh` | Read state_root from each node's witness DB, assert identical, print |
| `stagef_env.sh` | Sourced by other scripts; exports BASE_DIR, ports, pubkey file paths |
| `tests/test_undelegate.sh` | DELEGATE → UNDELEGATE → assert state_root 3/3 |
| `tests/test_claim.sh` | (TODO) CLAIM_REWARD → assert state_root 3/3 |
| `tests/test_stake.sh` | (TODO) STAKE (new validator) → assert state_root 3/3 |

## Design notes

- All scripts `set -euo pipefail` — fail fast on any error.
- PIDs written to `$BASE_DIR/pids.txt` so `stagef_down.sh` can
  reliably kill even if shell session is new.
- `BASE_DIR` path is written to `/tmp/stagef_current` so helper
  scripts can find the active run without env var plumbing.
- Each nodus-server logs to `$BASE_DIR/node$N/nodus.log`. Check these
  when a test fails — state_root divergence usually leaves a trail.
- Committee size is 3 (quorum 2) — minimum viable BFT, fastest spawn.
  Production runs 7; the consensus-breaking bugs reproduce fine at 3.

## When to add a new test

Any new consensus-affecting TX type, wire format, or state mutation
should get a test here BEFORE landing in production. Template:

```bash
#!/usr/bin/env bash
set -euo pipefail
. "$(dirname "$0")/../stagef_env.sh"

submit_tx ...
wait_for_block_advance
stagef_diff "after-TX"
```

## Status (2026-04-19)

**Working:**
- Full isolation from production (HOME override + DNA_NO_FALLBACK env).
- 7-node bring-up / teardown.
- Fresh identity per node, port layout collision-free.
- state_root diff helper (`stagef_diff.sh`).

**Gap: genesis bootstrap.**
Stagef nodes boot with roster=1 (self only). BFT needs quorum=5 →
"consensus disabled (n=1 < 5)" when the client submits genesis.
Production solves this somehow (genesis was committed on every chain
this session) but the bootstrap path isn't trivially portable to a
localhost harness. Needs one more investigation session: trace how
production nodus-server goes from roster=1 → roster=7 during the
first genesis submit. Most likely the chain_def trailer in the
genesis TX SEEDS the roster before BFT runs, but the stagef path
isn't reaching that code.

Once genesis works, tests run end-to-end: submit TX → `stagef_diff.sh
"label"` → assert 7/7 state_root identical.

## Isolation fixes landed in this session

Three CLI code paths were silently pulling hardcoded production IPs
into any test HOME. All three now gated by `DNA_NO_FALLBACK=1`:

1. `dna_config.c` — default bootstrap on first-run + migration rewrite.
2. `nodus_init.c` Source 3 fallback merge.
3. `nodus_init.c` connect-failure retry path.
4. `nodus_init.c` `start_rtt_probe` — UNCONDITIONALLY probed all
   fallback IPs. This was the sneakiest source; it TOFU-upserted
   production fingerprints into known_nodes even when connect
   succeeded against stagef.

## Known limitations

- Requires cluster to start from genesis — cannot replay an existing
  chain into the harness.
- Single machine — can't catch true network-partition bugs.
- No slashing / long-running tests — TTL loops kick in after
  minutes. Tests should complete in < 60 seconds.
